# Testando navegação na prática com test case gerado

Nesta aula, é demonstrada a **execução prática de testes de navegação utilizando a skill E2E NavTest**, evidenciando como uma estrutura organizada de casos de teste permite validar o comportamento de aplicações web de forma sistemática e eficiente. O professor destaca que a automação de testes **não depende apenas da ferramenta utilizada, mas principalmente da forma como os testes são planejados e estruturados** dentro do *workflow* de desenvolvimento.

Inicialmente, a *skill* é executada para analisar toda a aplicação, realizando a exploração do código-fonte e identificando automaticamente elementos essenciais do sistema, como ***framework* utilizado, rotas, autenticação, fluxos de navegação e regras de acesso**. Nesse exemplo, o sistema é identificado como uma aplicação baseada em **Next.js**, com autenticação personalizada utilizando ***tokens* JWT**, armazenamento em ***Local Storage*** e uso de ***cookies HTTP Only*** para gerenciamento de sessão.

Durante essa análise, a ferramenta também identifica os diferentes **papéis de usuários** existentes no sistema, como usuário comum, administrador, suporte e analista, além de determinar quais funcionalidades cada perfil pode acessar. Essa identificação permite gerar **casos de teste específicos para cada tipo de usuário**, garantindo que a validação do sistema considere as permissões e restrições definidas pela arquitetura da aplicação.

Após a análise inicial, a *skill* gera automaticamente um **documento de casos de teste** contendo pré-requisitos, definição do ambiente de execução e uma *árvore de dependências* que representa a sequência lógica das ações necessárias para navegar no sistema. Essa estrutura permite visualizar os caminhos possíveis dentro da aplicação, como **registro de usuário, login, recuperação de senha, acesso ao dashboard** e execução de operações relacionadas ao gerenciamento de vídeos e pastas.

Em seguida, é realizada a execução automatizada dos testes de navegação utilizando o ***Playwright CLI***, no modo visual, permitindo acompanhar em tempo real as interações realizadas pela ferramenta. Durante a execução, o sistema cria usuários, realiza autenticação, navega entre páginas, executa operações de criação e exclusão de registros e valida o comportamento da interface em diferentes cenários. Esse processo demonstra como a automação pode **simular o comportamento real de um usuário** dentro da aplicação.

Também é observado que a ferramenta registra automaticamente **evidências das execuções** realizadas, como capturas de tela em caso de falha. Em um dos testes, por exemplo, ocorre um erro durante a transcrição de um vídeo, possivelmente devido à ausência de áudio no arquivo utilizado. A ferramenta registra essa falha, documenta o comportamento observado e mantém o histórico das ações executadas, permitindo que o desenvolvedor investigue a causa do problema posteriormente.

Outro comportamento relevante demonstrado é a **capacidade da ferramenta de adaptar a execução dos testes** quando determinados elementos da interface não são encontrados. Em alguns cenários, a ferramenta tenta acessar funcionalidades alternativas ou executar chamadas diretas à API para completar a validação. Esse comportamento evidencia a flexibilidade das ferramentas de automação, mas também reforça a necessidade de **revisar os resultados gerados** para garantir a precisão dos testes.

Ao final da execução, o **relatório de testes é atualizado** com o status de cada cenário, indicando quais testes foram concluídos com sucesso, quais foram parcialmente executados e quais apresentaram falhas. Essa atualização permite acompanhar o progresso da validação do sistema e identificar rapidamente pontos que exigem ajustes ou melhorias. O professor destaca que o ideal é **atualizar o relatório de forma incremental**, registrando o resultado de cada etapa conforme ela é concluída.

Por fim, é reforçada a importância de **revisar continuamente os casos de teste** e realizar ajustes sempre que necessário. Embora as ferramentas automatizadas sejam capazes de gerar testes e identificar problemas, o **papel do desenvolvedor continua sendo essencial** para validar os resultados, interpretar falhas e aprimorar o processo de testes. A principal mensagem da aula é que a automação de testes de navegação deve ser baseada em uma **estrutura clara e iterativa**, capaz de evoluir conforme o sistema se torna mais complexo.
