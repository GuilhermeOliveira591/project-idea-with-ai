# Definição e categorização de testes

Qual é o grande problema em relação a testar com IA? A primeira coisa básica — e é básica, mas não é — é a **definição clara sobre o que são testes de unidade, integração e end-to-end**. Pode parecer uma coisa extremamente básica, mas acreditem: essas interpretações mudam muito de desenvolvedor para desenvolvedor e de empresa para empresa. Digo isso porque conheço desenvolvedores fantásticos, com mega experiência, e se eu perguntar para eles o que é um teste de integração, cada um vai falar algo diferente. Existem **zonas cinzentas** no meio de toda essa história, até mesmo em testes de unidade.

A grande questão é: você precisa dar clareza muito grande para a IA sobre o que é cada tipo de teste. Porque se eu, como desenvolvedor, entendo que um teste *end-to-end* é X e outro desenvolvedor entende que é Y, isso significa que, se não estou em concordância com outro desenvolvedor, **provavelmente a IA também pode ter essa ambiguidade**. Na cabeça dela, ela pode estar fazendo um teste *end-to-end*, mas, na realidade, ela está fazendo um teste de integração. Precisamos tomar cuidado com esses aspectos.

Outro aspecto que devemos levar em consideração é o **contexto**. Provavelmente, se você estiver em uma aplicação que é 100% *back-end* e pedir para fazer um teste *end-to-end*, a IA vai se comportar diferente do que em um projeto que tem *front-end*. Normalmente, um projeto com *front-end* terá testes *end-to-end* envolvendo interface gráfica. E o teste de integração? Será que tem? O que ela entende como integração? O que entende como unidade? Se você parou um pouco para raciocinar, perceba que muita gente pode ter essa dúvida, e a IA também terá. Ela vai falar o que acha.

Vou trazer alguns pontos para você considerar ao instruir a IA. Quando falamos de **testes de unidade**, a ideia básica é testar uma classe, um método, normalmente onde não há dependências. O grande ponto é que, muitas vezes, você pede para a IA fazer um teste de unidade e, para não ter dependências externas, ela vai criar um *mock*, uma classe *fake*, um serviço *fake*, para conseguir testar apenas o componente isolado. O problema é que, às vezes, você está pedindo um teste de unidade para algo tão granular, tão bobo, que além de criar esse teste inútil, ela ainda cria um *mock* para validá-lo. Assim, você começa a ter **testes inúteis** e arquivos que não agregam nada ao sistema.

Se você trabalha com *front-end*, precisa tomar ainda mais cuidado. Quando pede para testar um componente do React, por exemplo, a coisa vai para outro nível. Ela começa a testar pedacinho por pedacinho, *layer* por *layer*, e quando você vê, tem **dois mil testes criados para testar uma página de Hello World**, que não faria nenhum sentido testar.

Agora, sobre **testes de integração**: como isso pode ser visto por diversos desenvolvedores? Teste de integração pode ser integração entre classes e componentes do sistema — um componente chama outro diretamente, sem necessidade de *mockar* eles, testando de forma integrada. Por outro lado, conheço gente que faz teste de integração sem usar o banco de dados, testando apenas componentes internos. Para testar banco de dados, usam `SQLite` em memória para os testes passarem mais rápido. Mas muita gente utiliza `PostgreSQL` e roda os testes de integração batendo direto no `PostgreSQL`, embora às vezes fique mais lento. O grande problema é que `SQLite` não é totalmente compatível com `PostgreSQL`, e há problemas de índices, etc. **Como a IA vai interpretar esse teste de integração?**

Outra coisa: já vi IA falando que teste de integração é quando você testa **integrações externas**, como um *gateway* de pagamento. Ou seja, ela vai querer acessar sistemas externos para testar essa integração. O que faz sentido, inclusive, ser um teste de integração. Agora, eu posso ter um teste de integração entre *back-end* e *front-end* — executo uma ação no *front-end* e tenho uma reação no *back-end*, integrando os dois. Acreditem, para muita gente isso seria um teste *end-to-end*, mas há quem considere teste de integração. Não estou aqui para julgar o que é certo ou errado. Meu ponto é trazer para vocês que existem interpretações diferentes, e você não sabe qual será a interpretação da IA em relação a isso.

**Testes end-to-end** também têm interpretações variadas. Fazer uma chamada completa em uma API pode ser um teste *end-to-end*. Ou fazer um teste que chama um serviço externo: se chamo uma API e essa API chama outra API externa, essa chamada externa é um teste *end-to-end*? Se essa outra API externa não estiver no ar, meu teste *end-to-end* falhou mesmo eu não sendo dono dela? Onde ele acaba? Se abro o *browser*, executo uma ação que deve chamar a API, isso é um teste *end-to-end*? Se verifico como está o *layout* da página ou se a navegação está funcionando, também é um teste *end-to-end*?

O perigo de tudo **não é o nome que você dá para os testes**. Se você quiser chamar de teste de unidade e, na minha cabeça, o seu teste de unidade é o meu teste de integração, não tenho problema nenhum com isso. Mas a IA vai ter um problema com isso. Você corre um perigo muito grande: **a falta de definição clara sobre o que cada categoria de teste representa**. Isso pode gerar algo catastrófico. Ou ela gera dois mil testes, que vão gastar tempo na esteira de *CI/CD*, demorando 50 minutos para fazer um *build*, ou ela sai disparando testes de tudo quanto é lado para serviços externos, ou começa a fazer testes de UI com um monte de *browser*, gastando muito mais *tokens* do que você gostaria.

**Antes de começar a desenvolver, defina quais são os tipos de teste.** Parece que tudo que falo é: *"Quando esse cara vai começar a desenvolver?"* Pensem: desenvolver software vocês já sabem. O problema aqui é **como vocês dão a base para a IA desenvolver o software junto com vocês**. Por isso o que estou passando é muito importante. Sem essa base, você vai ficar como um desenvolvedor ordinário, pedindo coisas para a IA fazer e passando raiva. Essas coisas mudam o jogo na hora que você está desenvolvendo com IA.
