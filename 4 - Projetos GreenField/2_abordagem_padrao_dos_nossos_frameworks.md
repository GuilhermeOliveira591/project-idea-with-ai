# Abordagem padrão dos nossos frameworks

Vamos falar sobre começar a desenvolver um projeto do zero. Estamos ali com o VSCode aberto, sem nada, e como eu inicio um projeto para trabalhar com IA? Passei por essas sensações diversas vezes, pensando *"o que eu faço agora? Chamo a IA para criar para mim?"* Mas temos alguns pontos que são meio óbvios.

Primeiro, posso ter a tecnologia ou framework com o qual estou acostumado. Normalmente, esses frameworks têm **scripts de inicialização**. Por exemplo, vou iniciar um projeto base com **Spring Boot, Next.js, Laravel**, etc., e ele já gera a estrutura básica do projeto com apenas um comando.

Outra coisa que tenho percebido muito é quando solicitamos que a IA crie nossa estrutura do zero. **Precisamos ter cuidado aqui.** Não sei há quanto tempo vocês estão praticando com IA, mas muitas vezes temos a sensação de que, se não estamos utilizando inteligência artificial para fazer alguma coisa, é porque estamos para trás ou tem algo errado. Começamos a ter aquele **FOMO**, aquele *Fear of Missing Out*, como se estivéssemos fazendo algo errado, como se todo mundo estivesse fazendo de um jeito e só nós estivéssemos fazendo do jeito antigo. Mas precisamos usar o bom senso. Se pedir para criar algo do zero pode demorar muito mais, provavelmente não vai valer a pena. Porém, sempre existem exceções, e no mundo real as coisas funcionam de forma diferente.

Uma abordagem mais interessante, e na qual vamos focar mais, é a **skill customizada** para seguir um determinado padrão da empresa. Mas vamos analisar as opções.

Se eu já tenho o framework que vou utilizar, com scripts e arquivos que inicializam meu projeto, aquele *scaffolding* inicial, para que vou pedir para a IA? Não significa que não vou usar IA para isso, ou posso até pedir para a IA rodar o comando para mim. Mas pedir para ela sair criando as coisas uma a uma não vai fazer sentido.

Obviamente, você precisa entender e conhecer o framework e a ferramenta que trabalha, para saber o que é recurso nativo que pode fazer sentido para o seu projeto e o que realmente precisará customizar. Se for trabalhar da forma mais *vanilla* possível e não precisar de muita customização, rodarei um comando e depois pensarei em como trabalhar com IA.

O que acontece ao longo do tempo é que, na maioria das vezes, essa estrutura ou esqueleto inicial vai mudar, e às vezes muda bastante ao longo do projeto. É aí que precisamos começar a pensar. Não quero partir do total zero, ou seja, rodarei um comando para gerar uma estrutura, mas e depois? **Precisamos equilibrar para não sair nem para um extremo nem para o outro.**

Outro ponto importante: **se o comando que você solicita para a IA for exatamente o mesmo que você digitaria no terminal e já sabe de cabeça, digite o bendito comando.** Tenho percebido, e assumo isso claramente, que alguns comandos básicos que eu estava extremamente acostumado a digitar no dia a dia, tenho esquecido e muitas vezes pedido para a IA fazer para mim. Em alguns momentos, sinto que estou fazendo um desperdício ridículo de token por preguiça. Algumas vezes, vejo que a IA faz algumas coisas um pouco melhor para mim. Por exemplo, comandos de *commit*: dou um `//commit`, ela olha todos os arquivos soltos, entende o contexto, cria o commit para você com uma mensagem decente, e depois, se você der `tab` e `enter`, ela faz o *push*. O *commit* é um comando interessante; o *push* você poderia dar, mas já está ali na sua frente. A tendência de hoje pedir para a IA fazer até um push e gastar token com isso pode ser muito grande, e acredito que isso cada vez será mais natural. Precisamos ficar ligados nisso.

Mas o principal ponto é em relação à abordagem de iniciar um projeto. Se está trabalhando com framework, usar aquele esqueleto inicial para começar o projeto, eu acho que isso não muda nada. Porém, perceberemos que **isso não resolve nosso problema com o desenvolvimento em relação à IA**.
