Eu sei que são muitos vídeos apenas para falar de um CLAUDE.md ou qualquer arquivo desse tipo. Mas não gastaria um capítulo inteiro e o seu tempo se eu não achasse que isso é muito importante. Vale a pena detalhar e ir mais a fundo porque é uma das principais coisas que podem guiar a evolução no dia a dia.

Agora, uma questão importante: como evoluir esse arquivo e de onde partir? Primeiro, se o seu projeto não tem absolutamente nada, como um CLAUDE.md, algumas ferramentas têm um comando /init. O init simplesmente explora o seu codebase e faz um resumo do sistema, trazendo a ideia principal. Isso é muito importante porque faz você não partir do zero. No entanto, esse init tem uma particularidade: ele não traz detalhes muito claros de como subir e descer o ambiente. Normalmente, ele fala um pouco mais da stack, mostra um pouco mais a estrutura de pastas e coisas do tipo. Ele é um ponto de partida, mas está muito longe de ser um arquivo que vai gerar um impacto tão forte se você o fizer de forma intencional.

Vou dar um exemplo. Vou criar um novo branch, remover o CLAUDE.md e o README. Agora, vou abrir o Claude e usar o comando /init. Ele vai analisar o codebase e trazer uma versão inicial. Enquanto roda, volto para explicar: quando você roda esse init, ele gera um esqueleto, dá as diretrizes iniciais, faz você não partir do zero, mas é muito pouco específico. A ideia é que você comece a seguir uma linha de raciocínio para organizar esse arquivo.

Essa primeira linha de raciocínio é o que mostrei logo no início do capítulo: quick start, inicialização, state, ou seja, como ele vai desenvolver, principalmente na parte de testes, e como ele vai saber resumir o trabalho. Para a parte de desenvolvimento, os testes já são importantes e vão ajudar muito. É aqui que entra testabilidade e feedback.

Só de você conseguir testar, verificar o estado, inicializar e ter algumas coisas básicas que fazem a IA continuar desenvolvendo de onde parou, ou que a ajudem a desenvolver uma feature de forma mais fluida, você já vai resolver 80% dos problemas bobos que encontra hoje com a IA. O mais interessante é que, às vezes, usamos um modelo extremamente caro e lento para fazer uma tarefa que um modelo mais barato, sem reasoning, conseguiria fazer simplesmente por conta desse arquivo de inicialização. Esses quatro pontos já resolvem muita coisa para fazer o sistema rodar.

Esse é o nosso ponto de partida. Infelizmente, não vejo muitas empresas trabalhando com isso de forma séria. E normalmente, quando trabalham, é de forma não tão séria, porque os próprios desenvolvedores ainda não entenderam o valor desses arquivos. Por isso que digo: esse arquivo precisa muito de code review. E a code review dele precisa ter uma gama de testes, para que o time realmente verifique se o comportamento da IA está adequado, porque esse arquivo vai impactar no processo de desenvolvimento de toda a equipe. Se a sua equipe quer ser mais produtiva, esse arquivo tem que ser o melhor possível.

Não é porque alguém fez uma mudança no CLAUDE.md que você bate o olho e já aprova. É preciso verificar se realmente funcionou, porque não é tão simples revisar esse arquivo. Às vezes, algo que estava muito bom deixa de funcionar simplesmente porque fizeram uma besteira grande ali. Não deixe esse arquivo voando de lá para cá sem cuidado.
Outra coisa interessante: se o arquivo ficou muito grande, crie seções em arquivos separados. Marque, grife, crie títulos fortes para pontos críticos. Isso é importante. E aqui, sincronize com as skills. Sempre que adicionar uma nova skill, verifique se há alguma instrução que não deve ser movida e vice-versa. Falaremos um pouco disso depois, mas skills são coisas que o agente carrega on-demand, ou seja, conforme o agente precisa, ele carrega uma determinada skill. Se ele consegue carregar isso sozinho, talvez esse conhecimento não precise mais estar no CLAUDE.md, pode estar dentro de uma skill.

Vamos falar sobre skill em outro momento, mas somente para perceberem: se hoje vocês já têm skills rodando nos projetos e essas skills estão sendo chamadas normalmente durante o processo de desenvolvimento, provavelmente vale um teste: remover alguns conhecimentos, algumas informações que estejam no seu arquivo de inicialização, deixar somente na skill. Não é bom ter conhecimento duplicado, porque muda num lugar e não muda no outro, e aí começamos a ter ambiguidade em todo o sistema.

A regra geral, se pudesse falar alguma coisa para vocês, é: o que deve ser carregado sempre fica no CLAUDE.md, AGENTS.md, GEMINI.md, etc. O que nem sempre precisa ser carregado a todo momento, ou pode ser esporádico — por exemplo, quando ele vai precisar testar de um jeito tão específico — talvez valha a pena fazer de outra forma. Quando for desenvolver algo, se preciso disso toda hora, coloco no arquivo principal. Mas a tentação é querer que ele saiba tudo para ficar mais fácil. E na real, não. Tudo significa basicamente nada, porque se você colocar tudo, o arquivo vai ficar muito grande. O que eu quero que ele carregue é o básico, o necessário para o meu processo de desenvolvimento ser fluido.

Vamos olhar aqui o init que fizemos, para vocês verem como ficou. Agora tenho o CLAUDE.md. Ele fornecerá orientação para o Claude Code. O Project Overview: é um sistema de processamento de plataforma, upload de vídeos até 2GB, automaticamente transcrito usando OpenAI Whisper, gerando sumários com tópicos e referências. Deu um resumo, mas não colocou que era WebAssembly, não colocou que começava em 500 MB. Comandos de desenvolvimento, testes, front-end, docker, estrutura, key data flow, testes. Ele já fez alguma coisa. Mas como estou usando Docker no ambiente e não subo o ambiente com virtualenv, isso já é um problema. Ele acertou bem a parte de arquitetura: o fluxo de dados, vídeo usa chunked upload, cliente extrai vídeo usando FFmpeg, back-end processa a transcrição e gera o top organizado, real-time update via WebSockets. Conseguiu pegar bastante coisa: testes de unidade, testes de integração, front-end. Trouxe bastante informação. Algumas informações não são tão acuradas, mas foi um comando e já fez a gente não partir do zero.

Isso não é um curso de Claude Code. Existem várias manhas que você pode fazer na hora de rodar o /init, mas isso fica para outra vez. O ponto que quero trazer é que esse arquivo deve evoluir ao longo do projeto e você, desenvolvedor, precisa tratá-lo como um dos principais arquivos do seu projeto. Não se engane com READMEs e manuais que você acaba colocando no projeto. Nem sempre a IA vai ler, e esses documentos não são feitos para a IA ler, são feitos para humanos lerem. São grandes, têm detalhes às vezes desnecessários. README não significa um CLAUDE.md, um AGENTS.md, um GEMINI.md. Se você quer ter mais precisão, você usa esse arquivo.
