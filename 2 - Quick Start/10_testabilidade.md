Continuando. Agora entramos em uma parte crítica: os testes. Se tem uma coisa que vai acontecer muito durante o processo de desenvolvimento é rodar testes. E se tem uma coisa que você não quer que a IA perca tempo é tentando entender como rodar os testes. Porque existem sistemas onde os testes rápidos rodam em memória no SQLite, como nesse exemplo. Existem casos onde alguns tipos de testes precisam rodar com o mesmo banco de dados que vai ser usado em produção, mas dentro do ambiente Docker da sua máquina. Em outras situações, o teste pode ser completamente manual. Isso precisa estar transparente para a IA. Ela não deve precisar varrer os arquivos para entender como rodar um comando de teste. Assim como na inicialização, pedir para rodar teste deve ser a coisa mais simples do mundo, sem que ela hesite.

Aqui nesse caso, tenho Python markers. Quando peço para rodar testes de unidade, ela sabe quais marcações usar. Para testes de integração, é outra marcação, e já está claro que para rodar integração é necessário banco. Quando os testes são lentos, existe uma forma específica de executá-los, e para pular os lentos, usa-se -n not slow. Isso muda completamente a experiência da IA ao desenvolver qualquer tipo de teste. Quando eu disser "rode o teste de unidade" ou "rode testes correlatos", ela saberá quais marcadores usar e como executar.

Também é importante saber quais dados já estão disponíveis no sistema. Nesse caso, temos banco de dados, client, test user, usuário não verificado, autenticação, usuários múltiplos, etc. Tudo o que está disponível para ela utilizar. E os helpers no conftest, como createTestUserData, createLoginData.

Rodar testes — a IA conseguiria inferir, principalmente se for Python com pytest ou Node com Jest. Mas nesse caso específico, quero garantir que a IA rode os testes dentro do container, apesar de já ter colocado o comando. Prefiro dar uma listinha muito simples de como rodar os testes com comando direto, para ela não ter nem que pensar: todos os testes, teste específico pelo nome, somente testes de unidade, somente testes de integração. Está aqui o comando. Ela tem a árvore de diretórios, caso queira saber mais. Nesse projeto, os testes estão rodando sem problemas, sem que a IA fique perdendo tempo procurando como executar.

No front-end, com Jest, o teste de ambiente, o módulo de alias resolve com @, o arquivo de setup, e fornece browser API mocks. Teste no front-end é um mundo completamente diferente. Se você pedir para a IA sair fazendo testes, ela pode criar testes para todos os componentes e você acaba com 50 mil testes testando coisas bobas. Por isso, existem alguns padrões sendo usados para os testes. E como rodar os testes no front-end? Importante: sempre incluir --watchAll false para evitar modo interativo.

Outra coisa: toda vez que eu estava fazendo teste end-to-end com o Playwright, ele estava gerando diretórios de report, que eu não quero. Então, coloquei o comando que ele vai rodar, bonitão para fazer o teste com o Playwright sem aparecer aqueles reports.

Não é nada de outro mundo. São coisas que você percebe no dia a dia. Se a IA perde tempo tentando fazer algum tipo de teste e você vê que ela não entendeu ou não funcionou, você adiciona no seu arquivo, salva, manda executar de novo e vê se resolveu. É um pouco de tentativa e erro, mas quando você acerta a linha, você não vai mais ter aquele tipo de problema.

Regras de teste: nunca use --coverage ou --cover-report. No Playwright, sempre use --report=list. Quando rodar? Todos os testes rodam com mudanças maiores, antes de PR, antes de uma tarefa específica. Testes específicos, focados somente nas mudanças durante a interação. Por que coloco isso? Porque às vezes você tem tantos testes que, quando você pede para fazer algo e testar, ela acaba rodando todos os testes do sistema. Você pode colocar uma regra simples: mencionar os arquivos modificados e rodar só os que foram alterados no momento. Antes de um PR, ou se a mudança foi muito grande, roda todos os testes. Como ela sabe se é grande ou específico? É inferência. Por isso é importante testar o comportamento. Do jeito que coloquei aqui, está funcionando perfeitamente. Talvez pudesse estar mais aberto, ou precisasse de exemplos de testes específicos, mas mesmo assim gerei guidelines.

O que não testar? Validação de esquema, CRUD simples (os testes de integração já vão cobrir), renderização de componentes, comportamentos do framework, utilidades triviais como formatar data ou formatar bytes. E o que testar? Critical user flows, tipo autenticação, upload, core CRUD, business logic com edge cases, locality, quotas, permissions, error handling, state management, edge cases. Edge cases normalmente são aqueles pontos que não são óbvios e ele vai poder testar.

Estou deixando claro para evitar testes desnecessários. Cheguei nisso porque percebi que ele estava fazendo testes muito bobos: testando validação do Pydantic, sendo que Pydantic já é uma biblioteca que já faz a validação, ou fazendo microtestes que só gastavam tempo e geravam arquivos desnecessários. Se você quiser, pode ser ainda mais específico.

Mais algumas regras: no máximo 10 testes por arquivo, no máximo 5 mocks por teste. Se o arquivo de teste tiver uma implementação maior que o arquivo de código, talvez esteja testando demais. Rode os testes relacionados a cada finishing task step. Por quê? Porque no meu workflow tenho um step de tasks. A IA sabe qual passo ela está no processo de desenvolvimento. Cada vez mais, essas coisas vão ficando específicas do projeto.
