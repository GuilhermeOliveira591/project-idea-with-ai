Outra coisa interessante é a Git Policy. Para commits, é conventional format. Coloquei git commit -m "short description" e descrição detalhada, se houver muitas alterações de arquivo. Coloquei um detalhe muito simples: nunca mude o git config, nunca rode comandos destrutivos como force push, hard reset sem solicitação explícita, nunca skip hooks, use amend somente se realmente for safe, e nunca dê force push na main, master ou coisas desse tipo. Por que coloquei isso? Foram basicamente regras de segurança, porque vi que a IA estava rodando alguns comandos meio perigosos no Git.

Existem formas de resolver isso sem estar no CLAUDE.md. Você pode trabalhar com hooks que, na hora de chamar uma ferramenta, rodam e travam antes daquele tipo de comando. Vamos falar disso inclusive. Muitas vezes, algumas coisas não precisam estar no CLAUDE.md, desde que você tenha maneiras de lidar de forma automatizada. Por exemplo, você pode ver que não coloquei nada de linting, porque existe outra forma de garantir toda essa parte sem mexer nesse arquivo, usando hooks.

Tem um outro ponto super importante: os pitfalls — aquelas pegadinhas, os gotchas que normalmente acontecem. Toda hora que estamos desenvolvendo, percebemos que a IA está fazendo uma besteira em determinado arquivo, ou fazendo aquele negócio errado sempre. Você quer dar uma direção para que ela evite aquele problema. Por exemplo, clique propagation em links, interactive elements inside Next.js Link must stop propagation. Tem exemplo de react-hook-form refs — você não pode fazer override disso. Quando você está criando um componente para wrapping HTML, você tem que suportar o onChange custom. Por que tem isso? Porque durante o desenvolvimento do projeto, esses tipos de erros estavam acontecendo. Uma das formas de resolver é simplesmente dar um exemplo para ele tomar cuidado.
