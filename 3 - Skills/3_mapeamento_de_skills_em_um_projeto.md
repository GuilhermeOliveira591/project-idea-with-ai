# Mapeamento de Skills em um projeto

Agora que já deu para entender a ideia e o porquê de existirem as skills, e somente para você saber, quem criou a ideia das skills foi a **Anthropic**, que lançou isso no Claude Code. Em seguida, todo mundo passou a suportar. Então, hoje, se você pegar o Codex, o GitHub Copilot, o Cursor e praticamente todas as ferramentas de desenvolvimento atuais, elas já suportam skills também.

Então, o que acontece é o seguinte: quando vamos começar um novo projeto, ou se você já tem um projeto atual e decide que, a partir de agora, vai utilizar esse projeto para trabalhar usando IA, você precisa entender o seguinte: se você não está usando skill nesse projeto, pode ser que esteja gastando tokens à toa, ou que não esteja aproveitando todo o potencial disponível na hora de desenvolver a sua aplicação utilizando IA.

Nós falamos bastante sobre `cloud.md` e `agents.md`, onde colocamos, de forma sucinta, tudo o que o agente precisa para se comportar corretamente. Mas isso é carregado imediatamente no momento em que o agente inicia a sessão. Agora, se pegarmos tudo isso e colocarmos dentro do `agents.md`, do `cloud.md` e assim por diante, o que vai acontecer? Tudo isso irá diretamente para a janela de contexto.

Então, quando estamos pensando e começando um projeto, além de toda a preparação que fazemos com um `cloud.md` e um `agents.md`, precisamos pensar onde vamos colocar skills no nosso projeto. E aí vem um ponto que muitas pessoas costumam levantar: só porque existe a possibilidade de utilizar skill, não significa que precisamos usar uma skill.

Tem gente que diz que você estaria se forçando a usar uma skill apenas para afirmar que o seu projeto utiliza skills. E a minha resposta para isso é diferente. Na realidade, provavelmente, se você não está usando nenhuma skill no seu projeto, o nível de assertividade que a IA terá para trabalhar nele será menor. Muito provavelmente, o seu projeto pode ter diversas skills, e é difícil imaginar, hoje em dia, um projeto de desenvolvimento de software que não precise utilizar algum tipo de skill.

E aí está o grande ponto: como muitos desenvolvedores não conhecem skills, ou já ouviram falar, mas pensam que não precisam disso, acaba ficando difícil até mesmo escolher qual skill deve ser incluída no projeto. Por isso, eu criei aqui um racional para você pensar, olhando agora para um projeto que você já tem, quais skills poderiam ou fariam sentido estar nele.

A dica que eu dou para você, e isso pode ser controverso novamente, é a seguinte: **você precisa de skills no seu projeto**. Eu quase duvido que exista um projeto de desenvolvimento de software que não precise ter algum tipo de skill. Então, se você não tem nenhuma, provavelmente está deixando, vamos dizer assim, dinheiro na mesa, ou tokens na mesa.

Então, qual é um dos racionais que podemos utilizar para pensar em relação a isso? Em relação aos tipos de software. E, para esse racional começar a fazer sentido, eu acredito que vale a pena agrupar alguns desses tópicos em vídeos separados, para que não fiquemos com vídeos muito longos aqui. Então, vamos seguir a partir disso.
