# Racionais para mapeamento

**Racional para fazermos um mapeamento.** E o que é esse racional? Eu estou querendo, na realidade, trazer para você uma forma de encontrar uma linha de raciocínio para conseguir exercitar sua mente e identificar quais skills você precisa para aquele projeto.

Então, a primeira coisa que normalmente você vai fazer é definir e entender o seu projeto. É um projeto *back-end*, *front-end*? É um projeto *mobile*? É um projeto *full-stack*? Ou seja, é um projeto que tem, por exemplo, uma pasta back-end, que contém todos os arquivos de back-end, e uma pasta de front-end, que contém tudo relacionado ao front-end.

Ou você pode ter um **monorepo**, por exemplo. Além disso, você pode ter projetos back-end e front-end, como um Next.js, por exemplo, que possui toda a parte de front-end, mas também tem um back-end por trás dele. Somente ao entender esse tipo de estrutura, você já começa a direcionar quais skills podem ser mais gerais, com aspectos focados em back-end, front-end ou full-stack.

Vamos chegar nessas skills, eu vou mostrar sites e dar exemplos, mas apenas para você pensar e gerar essa separação na sua mente. Uma vez que você analisou isso, precisamos considerar que existem **skills com habilidades amplas**, com características mais conceituais. O que isso significa? Significa que não são específicas necessariamente de um framework ou da execução de uma determinada ferramenta.

Ou seja, são habilidades que fazem com que você, de forma mais estratégica, tenha um conhecimento relevante para conseguir executar determinado tipo de tarefa. Por exemplo, *front-end design*. Isso é uma skill. Quando falo de front-end design, não estou necessariamente falando de React, Vue.js, de um componente específico ou de Tailwind ou qualquer tecnologia desse tipo.

O que estou dizendo é que, provavelmente, uma skill desse tipo vai trazer boas práticas para você trabalhar com front-end. E quais podem ser essas práticas? Você sempre vai precisar refletir antes de criar elementos na tela. Evitar conflitos de usabilidade ao estruturar interfaces. Avaliar se determinada solução realmente faz sentido antes de implementá-la.

Se você observar, são habilidades mais de alto nível. Algo como uma habilidade estratégica de um engenheiro de software ou de um arquiteto de software. Não necessariamente focada em detalhes de implementação, mas voltada para princípios. O software segue boas práticas? Ele está utilizando *SOLID*?

Ele está seguindo uma estrutura ou algum tipo de arquitetura conhecido? Ele tem conceitos de *Clean Code*? Ele está utilizando padronização no nome dos arquivos? Esse projeto tem testes automatizados? Então, são todas perguntas que uma skill pode fazer ou que nós podemos fazer.

E, baseado nisso, o que vai acontecer? Na hora que ele for desenvolver um software, ele vai dizer: eu tenho uma skill aqui de uma habilidade ampla que vai me ajudar a analisar um projeto com base em boas práticas. Então, muitas vezes, eu preciso de boas práticas para desenvolvimento back-end.

O que isso significa? Na hora que eu disser que preciso desenvolver um software back-end dessa forma, essa skill vai ser ativada e vai pegar todos esses aspectos importantes de desenvolvimento para colocar em prática. Então, isso são **habilidades amplas**, ou seja, não têm necessariamente relação com uma tecnologia específica, mas trazem conceitos por trás que ajudam nesses aspectos.

Depois vamos ver algumas dessas skills, apenas para que fique mais claro e você consiga visualizar melhor. E um outro ponto interessante são as **Guidelines**. São outras categorizações que podemos considerar em relação às skills. Por que estou falando de Guidelines aqui? Porque uma habilidade ampla pode dizer como analisar algo, como criticar determinado ponto e como fazer uma avaliação mais conceitual.

Agora, quando falamos de *Guidelines*, estamos falando muito de algo chamado **Guidance**, que é um termo que você vai ouvir cada vez mais em IA. O que é Guidance, na prática? É quando você guia algo para ser feito. Ou seja, se você tem uma Guideline, usando essa Guideline, você terá uma Guidance para executar alguma coisa.

Então, quando eu tenho, por exemplo, um *Web Design Guidelines*, um *Go Development Guidelines*, um *Python Development Guidelines*, um *Next.js Guidelines*, isso significa que existem pontos específicos que precisam ser seguidos na hora de trabalhar. Ou seja, quais são as regras que devem ser utilizadas para seguir um determinado tipo de convenção.

Essas Guidelines não trazem apenas boas práticas, mas atuam de forma mais afirmativa sobre como algo deve ser feito. Uma habilidade ampla pode questionar, sugerir e orientar. Já uma Guideline define de forma mais direta o caminho a seguir. Em alguns casos, podem existir Guidelines dentro de habilidades mais amplas, e isso não tem problema, mas existem situações em que a Guideline é muito específica para determinado aspecto.

Isso é muito importante, inclusive, para facilitar a IA a seguir exatamente aquilo que você está determinando. E, nesse contexto, o que eu já posso dizer para você? O projeto em que você está trabalhando provavelmente tem padrões de nomes de arquivos, problemas legados, decisões técnicas anteriores e convenções próprias.

Isso acontece muito em **software legado**, quando uma parte do sistema está sendo modernizada e outra parte continua em um formato diferente. Então, quando você cria uma Guideline específica para o seu projeto, dentro dessa skill, você orienta a IA a ser guiada por essas duas formas que existem.

Ou seja, você consegue direcionar, de forma geral, a IA a utilizar essas Guidelines para tomar decisões durante o desenvolvimento. Então, uma coisa é algo conceitual, como boas práticas de front-end. Outra coisa é dizer: o front-end agora deve ser feito dessa forma. Isso é algo mais afirmativo.
