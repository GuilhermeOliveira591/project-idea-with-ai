# Racionais para Linguagens, Frameworks e Ferramentas

A gente já fez um racional aqui. Meu projeto é back-end, front-end, mobile, full-stack. Ok. Dependendo de como isso funciona, isso vai ajudar a gente a tomar decisões sobre quais tipos de skills vamos ter. E você deve estar pensando: existe uma skill de software back-end e front-end juntos, algo como full-stack?

Não necessariamente. Podem existir esses tipos de skills, sim, mas uma coisa importante é que, quando você coloca isso na sua cabeça e consegue fazer esse tipo de definição, você passa a ter capacidade de criar uma skill específica para o back-end e para o front-end da sua empresa, do seu projeto. E essas coisas acabam mudando ao longo do tempo.

Ou mesmo assim: meu projeto é mobile. Então, normalmente, quando vou desenvolver para mobile, existem algumas coisas que todo mundo precisa saber para trabalhar com esse tipo de aplicação. Se eu vou trabalhar com front-end, existem aspectos que preciso considerar. Por exemplo, a **responsividade**. Já na parte full-stack, preciso conseguir lidar muito bem com a integração entre back-end e front-end.

Lembre-se: nem todas essas categorias que estou apresentando aqui são necessariamente skills que já existem para aquele determinado caso, mas podem ser skills que vão passar a existir no seu projeto a partir de hoje, porque você pode começar a criar as suas próprias skills. E, se você ainda não pensou nisso, vale a pena refletir, porque isso pode fazer com que a IA erre muito menos. O nível de experiência que você vai ter no desenvolvimento com IA pode ser completamente diferente.

Então, vamos lá. O próximo ponto depois de Guidelines são as skills que são **Language Oriented**. O que isso significa? Como eu faço algo específico em Python? Por exemplo, como eu trabalho na otimização de performance em Python? Você consegue perceber que posso ter Guidelines de boas práticas para trabalhar com Python.

Mas performance é um assunto muito específico da linguagem e, muitas vezes, em Guidelines, você não consegue aprofundar tanto nesse tema. Então, vamos imaginar que você tem uma aplicação e o seu objetivo agora é trabalhar especificamente em performance para verificar se tudo está funcionando bem. Nesse momento, você pode ter uma skill específica para resolver esse tipo de tarefa.

Por exemplo, **TypeScript**. Posso ter uma Guideline básica de boas práticas com TypeScript, mas, se eu quiser trabalhar com tipos avançados em TypeScript, ou seja, algo mais orientado à linguagem e com um nível maior de profundidade, posso utilizar algo como *TypeScript Advanced Types*. Isso seria outra skill que você pode utilizar.

Agora começa a ficar interessante também quando você pensa em **Framework Oriented**. O que isso significa? Obviamente, todo framework tem uma linguagem de programação por trás. Por isso, temos Guidelines de linguagem e podemos ter recursos específicos para trabalhar com uma linguagem, algo *language oriented*. Mas também temos o conceito de *framework oriented*.

E aí é quando começamos a descer o nível, por quê? Porque, além de eu ter que saber programar em Python, se eu for trabalhar com **Django**, que é um framework em Python, eu vou ter Guidelines para utilizar boas práticas com Python. Porém, o Django tem todo um padrão, uma convenção própria que você precisa conhecer. A organização das pastas, onde colocar os templates, como criar os models, como desenvolver a parte de API, como funciona a área administrativa e como criar esses componentes.

Então, o que acontece nesse caso? São conhecimentos específicos de um framework. Por exemplo, eu quero trabalhar com boas práticas de React quando estou desenvolvendo aplicações na Vercel. Nesse cenário, eu posso ter uma skill voltada para o framework React, contendo boas práticas específicas para trabalhar com ele.

Eu posso ter, por exemplo, **Java Spring Boot**. Ou seja, eu quero uma skill que faça com que o meu desenvolvimento em Java, utilizando Spring Boot, seja carregado e ajude a IA a trabalhar com mais eficiência, evitando, por exemplo, que ela precise fazer consultas externas para buscar detalhes do Spring Boot ou algo semelhante.

Você sabe quando começa a perceber que estão faltando skills no seu projeto? É quando você pede para a IA executar alguma tarefa e, de repente, percebe que ela está fazendo *fetch* na web, pesquisando informações. Ou seja, você solicita algo usando Spring Boot e, em seguida, ela consulta a internet porque não tem aquele conhecimento estruturado disponível.

Provavelmente, se ela tivesse uma skill adequada e isso fosse relevante para o contexto, ela não precisaria fazer essa consulta. Ela não precisaria renderizar páginas extensas ou executar múltiplas buscas. As informações já estariam organizadas dentro de uma skill. Então, uma orientação prática é a seguinte: **se a IA precisou navegar na internet para executar algo que você solicitou, vale a pena verificar se existe uma skill para isso**.

Porque, se existir, muitas vezes é mais eficiente utilizar essa skill do que fazer a IA executar buscas repetidas. Essa é uma prática que tenho utilizado com frequência e que tem ajudado bastante no dia a dia de desenvolvimento.

E aí nós temos as **Skills Tools Oriented**. Ou seja, sempre que existe uma ferramenta que você utiliza no seu fluxo de trabalho, é possível ter uma skill específica para ela. Por exemplo, *GitHub Actions Templates*. O que isso significa? Às vezes, eu quero criar uma GitHub Action para executar o meu pipeline de CI/CD.

E, muitas vezes, posso pedir isso para a IA, e ela consegue gerar essa configuração. Mas existem muitos templates diferentes para criar GitHub Actions, dependendo do cenário. Em alguns casos, eu quero apenas executar um push. Em outros, quero rodar um CI, trabalhar com Docker ou configurar um code review automático com IA.

Novamente, são skills focadas em ajudar você a aproveitar melhor a IA para executar tarefas baseadas nas ferramentas utilizadas no dia a dia. Isso é muito importante. Inclusive, vou mostrar na prática uma outra skill orientada a ferramentas que pode ajudar bastante nesse processo.

Essa skill é relacionada ao **Playwright**. Não sei se você utiliza Playwright para navegação, testes de front-end ou automação durante o desenvolvimento. Mas existe uma skill específica para Playwright que pode ajudar muito, evitando que a IA precise fazer buscas externas ou utilizar constantemente servidores MCP relacionados a essa ferramenta.

Isso pode fazer com que você consuma menos tokens e tenha um fluxo de trabalho mais eficiente. Então, esse é um racional para o mapeamento. Agora, para não ficarmos apenas na teoria, eu quero mostrar, navegar e apresentar algumas ideias práticas relacionadas a essa parte de skills.
